Subdomain Takeover
Without further ado, let's get into our nest egg problem, the "Sudomain Hijack" problem. You had to take over test.hackedctf.com, which is exhibited by hosting content at that domain. It was a relatively simple one, largely predicated on knowledge of AWS S3, a storage service run by Amazon Web Services. It's largely known as a storage service, but it also works as a static website hosting service. As a result of this, you can point your domain to it and ccreate a bucket (effectively a top level folder, if you aren't familiar with AWS) by the name of that domain in AWS, hosting your content there.

Figure 1 displays approximately what you'd expect to see when you visited test.hackedctf.com in your browser. This made it easy. Creating a bucket by the name of test.hackedctf.com showed approximately what you see in Figure 2. Once you upload a file, in our case "tips.html", set it to the homepage, set the bucket to be a public static web hosting bucket, and the file (in AWS referred to as an object), you have enough to prove control.

Web Admin

Web Admin was compounded on the prior problem involving the shell we got in Web User. It involved a misuse of the system() call. In /home/leo/ on the system, the only user with a home directory, you found a run binary with setuid set and root ownership. When it executes, it executes as root. Any static analysis at all, which we did by base64'ing the file a la base64 /home/leo/run, copying the base64 of the file, then decoding it back into the file and throwing it in our team's comms channel, reveals it simply ran system("cat /etc/passwd") in C. This is not a new vulnerability, and I've seen it in a CTF when they made Twitch collectively exploit a binary in a "Twitch plays" fashion (the chat votes on the keystrokes), but the idea rests on the fact that "cat"'s location is decided by the PATH variable, which is under the user's control. Running export PATH=/tmp:$PATH means if you put a valid executable named /tmp/cat (must chmod +x to make it executable). I made /tmp/cat a one line shell script, first running ls /root to reveal a flag.txt file, then doing cat /root/flag.txt. The issue here is that you just broke cat to fix this. I caused an infinite loop on multiple occasions and kept having to kill my session and re-exploit. Ultimately, creating a one line shell script in /tmp/cat and invoking /home/leo/run results in the execution of /tmp/cat and as such, reading the flag.txt to the shell.

Git User 2

Git User 2 was interesting, because it was both a Gitlist server at port 80 and a Jenkins server at port 8080. I should disclose we used our (only) hint on this challenge. Gitlist was running a vulnerable version (0.4) and was succeptable to a no-auth RCE. The exploit code is available at the bottom of the blog (safe to click) here. The biggest pain is you have to consider the system, and how the exploit makes assumptions on infrastructure. It assumes you're attaching /var/www/gitlist, but we're attacking /var/www/html. Setting the cache to /var/www/html gives us a simple PHP shell, and if we change our base64 payload to a more complex PHP shell, which we spent too much time on, we're granted that. We then used that shell to find the flag and read it.
